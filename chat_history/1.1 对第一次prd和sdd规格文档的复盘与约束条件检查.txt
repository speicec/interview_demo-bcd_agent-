  二、缺失的任务约束（按优先级排列）

  阻塞级 —— 不解决会导致交付失败

  1. 缺少 Node.js 运行时可达性约束

  当前全部路径使用 node --experimental-vm-modules + TypeScript
  裸导入。package.json 中 "main": "./src/index.ts" 意味着必须由支持 TypeScript
  原生执行的环境运行（如 tsx、ts-node、Vite）。但 server workspace 连
  package.json 都没有，不存在可运行的入口。设计文档里没有任何地方写明
  如何从零启动服务。

  建议约束：每个 workspace 必须包含 "dev": "..." 脚本，且在一个干净的 git clone
  → npm install → npm run dev 后必须能启动。

  2. 缺少 API 契约测试约束

  spec 文件定义了 6 个 API 端点行为（/api/analyze, /api/chat/stream, /api/chat,
  /api/chat/:id/history, /api/batch/analyze），但 tasks.md
  中没有对应的契约测试任务。4.1-4.5 只要求"实现路由"，没有"验证契约"。

  建议约束：每个 API 端点必须有一个请求/响应快照测试，匹配 spec 中定义的
  Scenario。

  3. spec 与 tasks 之间缺少追溯矩阵

  查看 spec 中的 Scenario（约 30+ 个），无法追溯到哪些 task 覆盖了哪些
  scenario。例如：
  - blood-report-analysis/spec.md Scenario "Critical items shown first" →
  tasks.md 中无对应 task
  - ai-medical-chat/spec.md Scenario "LLM includes medical disclaimer" →
  tasks.md 中无对应验证 task
  - multi-indicator-correlation/spec.md Scenario "Only WBC elevated → no alert"
  → tasks.md 2.4 隐式覆盖但未显式追踪

  建议：为每个 task 添加 covers: 字段明确指向 spec scenarios。

  ---
  警告级 —— 不解决会导致质量风险

  4. 规则引擎测试覆盖不完整

  当前 ruleEngine.test.ts 缺失以下 spec 要求：
  - 病毒性感染模式（WBC↓ + LYMPH↑）
  - 正细胞性贫血（HGB↓ + MCV normal）
  - 大细胞性贫血（HGB↓ + MCV↑）
  - 血小板减少+凝血异常（spec 中无但代码实现了，需要测试）
  - 炎症三联征（WBC↑ + CRP↑ + ESR↑）
  - 缺铁性贫血可能性（HGB↓ + MCV↓ + RDWCV↑）
  - 未知指标代码的兜底行为
  - minItems: 1 校验场景（空 indicators 数组）

  5. 缺少非功能约束

  spec 明确要求但 tasks.md 中无对应 task：
  - JSON 校验 50 项指标 < 100ms → 需要性能基准测试
  - 完整分析链路 < 2 秒（不含 LLM）→ 需要端到端计时
  - SessionPool 100 上限 + 30min TTL → 需要并发/超时测试
  - WCAG AA 对比度 4.5:1 → 需要无障碍测试 task

  6. 缺少医疗场景特定的安全约束

  design.md 明确说"不提供诊断结论，仅提供健康提醒"，但代码中的
  interpretationRules.ts 存在措辞风险：
  - "需紧急就医排查严重感染" — "紧急就医"是否为诊断性建议？界限模糊
  - 没有自动化检查机制来审查 interpretation 文案是否符合免责声明要求

  建议约束：所有对外展示的文案必须经过 disclaimer:
  标签审查，禁止出现"确诊""建议用药""必须"等词汇。

  ---
  建议级 —— 改善但不阻塞

  7. batch-report-management 缺少空批次/单一报告边界约束

  spec 定义了"3 份报告"场景，但没有定义：
  - 仅 1 份报告时是否允许 batch 模式
  - 100 份报告时的性能预期
  - batch 模式下各报告之间是否允许共享 patient 信息

  8. ai-medical-chat 缺少中断/重连约束

  SSE 流式中断（网络断开、用户刷新页面）时：
  - 后端是否需要检测断连并释放资源？
  - 前端 reconnect 后是重放整条消息还是跳过？

  9. 缺少多 Indicator 码同名场景

  如果 JSON 中出现 [{code: "WBC", ...}, {code: "WBC", ...}]（同一个 code
  出现两次），当前实现取最后一个（无去重/冲突处理）。
